# Extra03 长图处理与切块策略专题

## 一、为什么长图是多模态系统里的典型难点

长截图、长网页、长票据、长合同页面，几乎是所有多模态应用迟早都会遇到的问题。

长图难，不是因为模型“完全看不懂”，而是因为长图天然会把几个问题叠在一起：

- 内容很多
- 小字很多
- 结构复杂
- 缩小后细节损失严重
- 直接整图输入成本高

这类问题如果处理不好，会直接让模型表现从“偶尔不稳”变成“系统性不准”。

## 二、为什么不能把长图直接缩小

这可能是长图场景里最常见、也最致命的错误。

如果你把一张超长截图直接等比例缩小，通常会发生：

- 字体变得不可辨认
- 表格线和边界消失
- 小图标和角标丢失
- 阅读顺序变得模糊

所以对长图来说，问题往往不是模型不够强，而是输入前就已经把信息毁掉了。

## 三、长图处理的三个基本策略

### 1. 整图粗读

先让模型对整张图做高层次摘要，例如：

- 这是一张什么页面
- 大致分成几个区域
- 页面主要任务是什么

这一层不追求细节，只建立全局认知。

### 2. 切块细读

把长图按区域或固定窗口切成多个小块，逐块分析：

- 每块有什么内容
- 是否有关键字段
- 哪块最值得继续深挖

### 3. 汇总归纳

把每块摘要重新汇总成最终答案，必要时加规则校验。

这本质上是一种“分层阅读”。

## 四、切块策略怎么选

### 1. 固定窗口切块

适合结构相对规整的长图，例如长网页、长后台列表。

优点：

- 实现简单
- 易于批处理

缺点：

- 可能把关键区域切断

### 2. 带重叠窗口切块

在相邻窗口之间保留一定重叠区域，用来减少边界信息丢失。

适合：

- 表格
- 长文档
- 关键字段跨块分布的页面

### 3. 版面感知切块

根据页面结构来切，例如按：

- 标题区
- 正文区
- 表格区
- 侧栏区

这种方式最合理，但通常也更依赖额外工具或版面分析能力。

## 五、一个实战案例：长截图报错分析

假设用户提供的是一张很长的后台报错截图，里面可能同时包含：

- 顶部导航
- 中间报错提示
- 底部日志信息

这时候建议工作流是：

1. 整图摘要，判断页面类型和大致结构
2. 按垂直方向切成 3 到 6 段
3. 对每段单独问“是否包含错误信息或关键提示”
4. 收集最相关的块，再汇总生成答案

这比直接问“整张图报了什么错”稳定得多。

## 六、一个实战案例：长文档字段抽取

如果目标是从长文档中抽字段，建议把任务拆成两类：

### 1. 定位类问题

例如：

- 哪一段包含合同金额
- 哪个区域有日期
- 是否有签章区域

### 2. 抽取类问题

在定位到局部区域后，再做字段提取。

这种“先定位、再抽取”的方式，比直接整图抽字段更稳。

## 七、一个实用的层级化 Prompt 思路

你可以把长图处理拆成三层 Prompt：

### 层 1：全局摘要

```text
请概括这张长图的页面类型、主要结构和核心内容，不需要细节。
```

### 层 2：局部切块分析

```text
请分析这一块区域，提取其中的关键文字、字段、报错信息或重要元素。
```

### 层 3：最终汇总

```text
根据整图摘要和各块分析结果，输出最终结论。如果部分信息不确定，请明确说明。
```

这个层级化方式很适合和 Agent 结合。

## 八、为什么长图特别适合做多模态 Agent

长图场景天然适合 Agent，不只是因为它复杂，而是因为它很适合“分步处理”：

- Step 1：判断页面类型
- Step 2：决定切块方式
- Step 3：筛选重点块
- Step 4：调用 OCR 或规则工具
- Step 5：汇总最终结果

所以你在第九章里看到的多模态 Agent 设计，在长图场景里尤其有价值。

## 九、工程上的几个务实建议

### 1. 保留原图

不要只保存缩略图。很多后续排查都需要回到原图。

### 2. 记录切块坐标

否则一旦模型说某块有问题，你没法回溯位置。

### 3. 汇总时保留证据来源

例如说明：

- 哪个块得出的结论
- 哪个字段来自 OCR
- 哪个判断来自模型推理

### 4. 允许“不确定”

长图场景比普通图片更容易出错，系统必须允许：

- 未识别
- 需人工确认
- 建议用户上传更清晰版本
