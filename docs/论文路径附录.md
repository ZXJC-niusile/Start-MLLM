# 论文路径附录：从教程到可发表工作

> 本附录是 [第十章第十一节](./chapter10/第十章%20学习路线与开源项目实战建议.md#十一给有想法想做成可发表工作的读者边界与最小闭环) 的扩展，面向希望把教程所学升级为可发表论文的读者。

## 一、论文不是"做更大"，而是"说清贡献"

很多初学者误以为论文 = 做更复杂的模型。实际上，论文的核心是**可检验的贡献陈述**：

- 你发现了什么问题？
- 现有方法在这个问题上哪里不够好？
- 你的方法为什么更好？
- 你用什么实验证明了"更好"？

贡献可以是数据配方、推理策略、工具编排、评测协议——**不必一开始就是新架构**。

## 二、读论文：三遍法

### 第一遍（10 分钟）：抓骨架

- 读标题、摘要、结论
- 看图和表（尤其是主实验表和消融表）
- 回答问题：这篇在解决什么问题？方法属于哪一类？效果提升多少？

### 第二遍（1 小时）：抓细节

- 读方法章节，理解核心创新点的数学/算法描述
- 读实验设置，确认数据集、基线、评价指标
- 回答问题：如果去掉这个方法的核心组件，性能会掉多少？（这就是消融）

### 第三遍（半天到一天）：抓复现

- 找到开源代码（如果有）
- 跑通官方 demo
- 尝试修改一个超参，观察输出变化
- 回答问题：我能在这个基础上做什么改进？

**对 MLLM 初学者，建议的起步阅读清单**：

| 论文 | 读的重点 | 为什么先读它 |
|---|---|---|
| LLaVA（2023） | Connector + 指令微调的范式 | 理解现代 VLM 的 baseline |
| Qwen-VL / Qwen2.5-VL 技术报告 | 国产最强开源 VLM 的设计决策 | 中文场景、可本地运行 |
| BLIP-2（2023） | Q-Former 轻量连接 | 理解"冻结两边只训中间"的范式 |
| MiniGPT-4（2023） | 单层 MLP 也能接视觉 | 极简 Connector 的上限 |
| DPO 原始论文（2023） | 偏好优化的数学推导 | 理解为什么 DPO 能替代 PPO |
| POPE / HallusionBench（2023-2024） | 幻觉评测设计 | 理解"怎么测幻觉"比"怎么治幻觉"先 |

## 三、复现一个 baseline 的具体流程

复现是论文工作的起点。如果你连 baseline 都跑不通，后面的改进就是空中楼阁。

### Step 1：选 baseline

- 选和你场景最接近的**一个**开源模型（不要贪多）
- 确认它有官方代码、有预训练权重、有评测脚本
- 推荐起步：Qwen2.5-VL-3B / LLaVA-1.5-7B / MiniCPM-V-2.6

### Step 2：跑通推理

- 用官方代码跑一张你自己的图，确认输出合理
- 记录：模型版本、transformers 版本、PyTorch 版本、CUDA 版本

### Step 3：跑通评测

- 在官方支持的 1-2 个 benchmark 上复现论文报告的数字
- 如果复现不了，先排查：精度设置（fp16/bf16）、beam search vs greedy、prompt 模板
- **自己的数字和论文数字差距在 1-2 个点内算复现成功**

### Step 4：建立你的评测集

- 用第五章的 `eval_vlm_dataset.py` 格式，做 50-200 条**你自己场景**的数据
- 跑 baseline，记录失败模式（这是你的改进动机来源）

## 四、消融表怎么设计

消融（Ablation）是论文的灵魂，它回答"你的方法中每个部分有多大用"。

### 标准消融表结构

| 配置 | 指标 A | 指标 B | 说明 |
|---|---|---|---|
| 完整方法 | 85.2 | 72.1 | 你的最终方案 |
| w/o 组件 X | 82.1 (-3.1) | 68.5 (-3.6) | 去掉你提出的核心组件 |
| w/o 组件 Y | 84.5 (-0.7) | 71.8 (-0.3) | 去掉次要组件 |
| 基线方法 | 78.3 | 65.4 | 最简单的已有方案 |

### 消融设计的原则

1. **每个消融只改一个变量**：不能同时去掉 A 和 B，否则不知道是哪个的作用
2. **必须有基线对比**：否则审稿人不知道"85.2"算高还是低
3. **失败的消融也要报**：如果去掉某个组件性能反而提升，诚实地写，然后解释可能原因
4. **消融要有理论依据**：不要随机去掉模块，要解释"为什么这个模块应该有用"

## 五、LaTeX 模板入门

### 推荐模板

| 会议/期刊 | 模板来源 | 特点 |
|---|---|---|
| ACL/EMNLP/NAACL | ACL Anthology | NLP 顶会标准模板 |
| CVPR/ICCV/ECCV | CVF Open Access | CV 顶会标准模板 |
| arXiv | 任意模板 | 预印本，无格式要求，但建议用标准模板 |
| IEEE TPAMI / TMM | IEEE 模板 | 期刊，审稿周期长但认可度高 |

### 最小可运行 LaTeX 示例

```latex
\documentclass[11pt]{article}
\usepackage[UTF8]{ctex}  % 中文支持
\usepackage{graphicx,booktabs,amsmath}

\title{你的论文标题}
\author{你的名字}
\date{\today}

\begin{document}
\maketitle

\begin{abstract}
用 150-250 字说清：问题、方法、实验结果。
\end{abstract}

\section{Introduction}
\begin{itemize}
    \item 背景：MLLM 在 X 场景很重要
    \item 问题：现有方法在 Y 上不够好
    \item 方法：我们提出了 Z
    \item 贡献：1)... 2)... 3)...
\end{itemize}

\section{Method}
写清你的方法，用公式 + 算法伪代码 + 示意图。

\section{Experiments}
\subsection{Setup}
数据集、评测指标、基线、实现细节。

\subsection{Main Results}
放主实验表。

\subsection{Ablation Studies}
放消融表。

\section{Conclusion}
总结 + 局限 + 未来方向。

\bibliography{references}
\end{document}
```

### 推荐的 LaTeX 工具链

- **Overleaf**（在线）：免配置，适合协作，免费版够用
- **TeX Live + VS Code + LaTeX Workshop**（本地）：更灵活，适合大项目
- **参考文献管理**：Zotero + Better BibTeX 插件，一键导出 `.bib`

## 六、arXiv 投稿流程

### 1. 注册账号

访问 https://arxiv.org/，用机构邮箱注册（更容易通过 endorsements）。

### 2. 准备文件

- `.tex` 主文件 + `.bib` 参考文献 + 图片文件夹
- 或者直接用 Overleaf 的 "Submit to arXiv" 功能导出 zip

### 3. 填写元数据

- Title：准确、不夸张
- Authors：按贡献排序，通讯作者标星
- Abstract：和论文摘要一致
- Comments：可写"Accepted to XXXX 2025"或"Under review"
- ACM class / MSC class：选最接近的类别

### 4. 上传和编译

arXiv 用 TeX Live 编译，注意：
- 不要包含 `hyperref` 的 `colorlinks`（arXiv 处理会出问题）
- 图片用 `.pdf` 或 `.png`，避免 `.eps`
- 字体要嵌入，不然 arXiv 会报错

### 5. 提交后

- 通常 1-2 个工作日放出
- 放出后可以更新（replace），但旧版本仍会保留
- 建议先在匿名期投出去，收到反馈后再决定是否投会议

## 七、Rebuttal 模板与策略

如果投了会议（如 ACL/CVPR），收到审稿意见后需要写 rebuttal。

### Rebuttal 的结构

```
Dear reviewers and AC,

We thank all reviewers for their constructive feedback. 

## Reviewer #1

Q1: [粘贴审稿问题]
A1: [逐条回答，用"We clarify that..." / "We have added..." / "We respectfully disagree because..."]

## Reviewer #2
...

## Summary of changes
[用 bullet 列出论文修改点，方便 AC 快速定位]
```

### Rebuttal 的写作原则

1. **先谢再答**：永远先感谢审稿人，再回答问题
2. **逐条对应**：用 Q1/A1 格式，不要混着答
3. **用数字说话**：如果审稿人质疑实验，补充数字比解释更有力
4. **诚实面对局限**：如果审稿人指出的问题确实存在，承认并说明未来工作
5. **不要攻击性语言**：即使审稿人明显错了，也要委婉表达

### 给初学者的会议策略

1. **第一年**：先投 workshop（如 CVPR/ACL workshop），门槛低、反馈快、容易中
2. **第二年**：投主会，但选 "Findings" 或短论文（short paper）轨道
3. **第三年及以后**：再冲 oral / long paper

## 九、从教程到论文的最小时间线

| 周数 | 任务 | 产出 |
|---|---|---|
| 1-2 | 读完本教程前 9 章，跑通第七、八、九章代码 | 可运行的 Demo + 评测 JSONL |
| 3-4 | 选定一个细分方向，读 5-10 篇相关论文 | 文献笔记 + gap 分析 |
| 5-6 | 复现 1 个 baseline，在自己的数据上跑通 | 复现报告 + 失败模式记录 |
| 7-8 | 提出改进，实现并实验 | 实验结果 + 消融表 |
| 9-10 | 写论文初稿（用 LaTeX 模板）| 完整论文草稿 |
| 11-12 | 导师/同行审阅，修改，投稿 | 投稿确认邮件 |

**关键提醒**：不要等"完美"才投稿。审稿人的反馈往往比你自己闷头改更有价值。

## 十、伦理与合规

- **数据合规**：如果数据包含人脸、证件、医疗信息，需确认已获得授权或已脱敏
- **模型许可**：商用需确认模型许可证（部分 Qwen 版本、Llama 有商用限制）
- ** baseline 引用**：务必正确引用你对比的所有 baseline 论文
- **可复现承诺**：投稿时声明代码和数据将在录用后公开（并真的做到）

## 十二、本章小结

- 论文的核心是**可检验的贡献陈述**，不是"做更大"
- 读论文用三遍法，复现 baseline 是第一步
- 消融表是论文的灵魂，每个消融只改一个变量
- 从教程到第一篇论文的最小时间线约 12 周
- 先投 workshop 或 findings 建立信心，再冲主会 oral
