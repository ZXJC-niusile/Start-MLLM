# 第十章 学习路线与开源项目实战建议

## 一、学完这份教程后，你下一步该做什么

如果你已经读完前面章节，并且跑通了代码，那么现在最重要的事情不是继续收集更多概念，而是把知识沉淀成自己的工程能力。

最推荐的方式有三种：

1. 继续扩写这个仓库
2. 做一个小型多模态应用
3. 针对一个细分方向持续深挖

## 二、一条务实的四阶段学习路线

### 第一阶段：理解主结构

目标：

- 说清楚视觉编码器、对齐层、LLM 的关系
- 看懂主流 VLM 的架构图
- 能解释为什么同样是多模态模型，表现会差很多

### 第二阶段：跑通最小实践

目标：

- 本地跑一个多模态模型
- 能通过 OpenAI 兼容接口发图文消息
- 做一个可交互 Demo

### 第三阶段：做场景化项目

你可以任选一个方向：

- 截图问答助手
- 文档抽取助手
- 商品图文案助手
- 图片批注或审核助手

### 第四阶段：做系统化工程

进一步加入：

- 模型路由
- OCR 专用工具
- 结构化输出
- 评测集
- 观测与日志

## 三、如果你要把这个仓库继续做大，建议怎么扩展

下面这条扩展路线比较自然：

### 方向 1：补理论章节

可以新增：

- 文档智能专题
- 视频理解专题
- 音频与语音多模态专题
- Grounding 与视觉定位专题

### 方向 2：补代码专题

可以新增：

- LoRA 微调脚本
- benchmark 脚本
- 多模型路由服务
- OCR + VLM 融合工作流

### 方向 3：补案例专题

可以新增：

- IDE 截图助手
- 面试题看图解析
- 票据抽取
- PPT/海报内容分析

## 四、做教程仓库时，最容易忽略但最重要的事

### 1. 保持目录稳定

教程仓库最怕目录经常重构。建议尽早固定：

- 根 README
- docs 章节
- code 示例
- extra 或 examples 扩展区

### 2. 一章只讲一个主问题

不要让单章既讲理论又讲所有工程细节，容易失焦。

### 3. 每个代码示例都要有清晰边界

读者应该能一眼知道：

- 这个脚本做什么
- 依赖是什么
- 怎么运行
- 适合改造成什么

### 4. 给读者一个继续贡献的入口

你可以在仓库里逐步加入：

- `TODO`
- `Good first issue`
- `examples/`
- `Extra-Chapter/`

这样仓库更容易变成真正的开源共创项目。

## 五、一个适合教程仓库的 Roadmap 示例

你可以直接把下面这组目标当成后续迭代计划：

- [ ] 增加中文 OCR 与文档理解专题
- [ ] 增加多图比较和长图处理专题
- [ ] 增加 LoRA 微调示例
- [ ] 增加 VLM 评测脚本
- [ ] 增加多模态 Agent Demo
- [ ] 增加视频理解入门章节
- [ ] 增加英文版 README

## 六、建议你开始形成自己的“模型判断框架”

真正的工程能力，不是会背模型名，而是能快速回答这些问题：

1. 这个模型的视觉侧强不强？
2. 它擅长自然图像、文档还是截图？
3. 它的输入格式和部署方式是否适合我的系统？
4. 如果回答不准，问题更可能出在什么地方？

当你能稳定回答这些问题时，你就已经从“调用者”进入“构建者”阶段了。

## 七、把这份仓库继续做成项目的三条路线

### 路线 1：教程型仓库

继续补章节、加配图、加练习、加案例，适合做长期内容积累。

### 路线 2：实验型仓库

重点放在：

- 评测脚本
- prompt 对比
- 模型横评
- Demo 快速验证

### 路线 3：应用型仓库

围绕一个明确场景持续迭代，例如：

- 截图助手
- 文档抽取
- 多模态客服
- 图像审核

这三条路线没有绝对高低，但最好一开始先选主线，不然仓库会很容易发散。

## 八、章末练习

1. 选择教程型、实验型、应用型三条路线中的一条，说明你为什么适合它。
2. 给 `Start-MLLM` 写一个你自己的 30 天迭代计划。
3. 列出你下一步最想补的 3 个文件或章节，并说明原因。
4. 如果你要把这个仓库开源给别人看，你希望别人一进来先看到什么？

## 十、本章小结

这份教程的最佳用法，不是读完就结束，而是继续把它演化成：

- 你的学习仓库
- 你的实验仓库
- 你的开源教程
- 你的项目作品集

如果你继续扩展这个项目，建议优先补“场景、代码、评测”三类内容，因为它们最容易把一个教程仓库做出厚度。
