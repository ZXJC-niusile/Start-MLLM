# 第十章 学习路线与开源项目实战建议

> **实战篇**（第 5～10 章）：评测、部署、推理与 Demo、Agent、学习路线收束。本章是实战篇的**路线收束**，不再引入新架构名词。

## 全书主线回扣

**理论篇（1～4 章）**把地图、视觉编码与对齐、生成架构和数据配方说清；**实战篇（5～10 章）**用同一批场景做离线评测（第 5 章）、把链路与观测搭稳（第 6 章）、落成脚本与 Demo（第 7～8 章）、再加感知与工具工作流（第 9 章）。本章把这些收成**可执行的迭代计划**，而不是再堆新概念。自检清单也可对照 [前言](../前言.md) 第九节「顺着正文走完，你大致能独立完成什么」。

**工程速记**：很多小团队一周里会在「补 20 条 JSONL → 跑 `eval_vlm_dataset.py` → 改预处理或换模型 → 再写进第六章日志字段」之间循环；第十章下面的阶段路线，就是把这种循环写进日历，而不是只记在脑子里。

## 一、学到这里，下一步该做什么

如果你已经读完前面章节、也跑通过代码，接下来最重要的不是继续囤概念，而是把这些内容沉淀成自己的工程能力。

最推荐的方式有三种：

1. 继续扩写这个仓库
2. 做一个小型多模态应用
3. 针对一个细分方向持续深挖

## 二、一条更务实的四阶段路线

### 第一阶段：理解主结构

目标：

- 说清楚视觉编码器、对齐层、LLM 的关系
- 看懂主流 VLM 的架构图
- 能解释为什么同样是多模态模型，表现会差很多

### 第二阶段：跑通最小实践

目标：

- 本地跑一个多模态模型
- 能通过 OpenAI 兼容接口发图文消息
- 做一个可交互 Demo

### 第三阶段：做场景化项目

任选一个方向先做都可以：

- 截图问答助手
- 文档抽取助手
- 商品图文案助手
- 图片批注或审核助手

### 第四阶段：做系统化工程

进一步加入：

- 模型路由
- OCR 专用工具
- 结构化输出
- 评测集
- 观测与日志

## 四、做教程仓库最容易忽略的事

### 1. 保持目录稳定

教程仓库最怕目录频繁重构，最好尽早固定：

- 根 README
- docs 章节
- code 示例
- extra 或 examples 扩展区

### 2. 一章只讲一个主问题

不要让单章既讲理论又讲所有工程细节，容易失焦。

### 3. 每个代码示例都要有清晰边界

读者应该能一眼知道：

- 这个脚本做什么
- 依赖是什么
- 怎么运行
- 适合改造成什么

### 4. 给读者一个继续贡献的入口

仓库里可以逐步加入：

- `TODO`
- `Good first issue`
- `examples/`
- `Extra-Chapter/`

这样仓库更容易变成真正的开源共创项目。

## 六、建议你尽早形成自己的“模型判断框架”

真正的工程能力，不是会背模型名，而是能快速回答这些问题：

1. 这个模型的视觉侧强不强？
2. 它擅长自然图像、文档还是截图？
3. 它的输入格式和部署方式是否适合我的系统？
4. 如果回答不准，问题更可能出在什么地方？

当你能稳定回答这些问题时，你就已经从“调用者”进入“构建者”阶段了。

## 七、把仓库继续做成项目的三条路线

### 路线 1：教程型仓库

继续补章节、加配图、加练习、加案例，适合做长期内容积累。

### 路线 2：实验型仓库

重点放在：

- 评测脚本
- prompt 对比
- 模型横评
- Demo 快速验证

### 路线 3：应用型仓库

围绕一个明确场景持续迭代，例如：

- 截图助手
- 文档抽取
- 多模态客服
- 图像审核

这三条路线没有绝对高低，但最好一开始先选主线，不然仓库会很容易发散。

## 八、章末练习

**动机**：收官要把九章收成你自己能执行的迭代主线；练完你清楚接下来 30 天优先补场景、评测还是部署，仓库不会无限发散。

### 必做（约 15 分钟）

1. 选择教程型、实验型、应用型三条路线中的一条，说明你为什么适合它。

### 进阶（约 40 分钟）

1. 给 `Start-MLLM` 写一个你自己的 30 天迭代计划（按周拆即可）。
2. 列出你下一步最想补的 3 个文件或章节，并说明原因。

### 挑战（1～2 小时）

1. 如果你要把这个仓库开源给别人看，写一段「首页 3 秒能看懂的价值」：新人应该先点哪三个链接、第一周建议完成哪三个产出（可对齐 [前言](../前言.md) 第九节能力清单）。

## 十一、给「有想法、想做成可发表工作」的读者：边界与最小闭环

本仓库的首要定位，是帮**只有 Python 基础**的人建立多模态系统的**可读、可跑、可验收**闭环（地图 → 数据格式 → 小评测集 → 推理/Demo/Agent）。走完正文，你通常已经能**独立实现自己想法的工程版**（换数据、换预处理、换评测维度、做小系统）。**发表论文**还要求额外一层：**可检验的贡献陈述、严格基线、对照与消融、可复现与相关工作**，这些无法只靠「读懂教程」自动获得，需要你在教程之外补文献与实验规范。

下面是一条**务实最小路径**，便于你自检「idea 是否已从直觉变成可写进论文段落的工作」：

1. **把想法写成可证伪的一句话**  
   例如：「在 X 类截图上，先做 Y 预处理再调用 VLM，比直接送原图能减少 Z 类错误。」贡献可以是数据配方、推理策略、工具编排、评测协议——不必一开始就是新架构。

2. **固定基线与变量**  
   同一模型、同一解码参数、同一评测集划分；只改你声称的那一环。基线至少包括：**直接 VLM**、**简单强基线**（如固定 prompt 模板、或公开小模型）。用 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 的 JSONL + `eval_vlm_dataset.py` 习惯，把样本量从「几条手感」扩到「能算比例、能分桶」——开放问答可辅以人工裁定或小规则辅助统计，但要在文中写清局限。

3. **消融（ablation）**  
   至少回答：去掉你的方法中的一步，性能或失败率如何变化？没有消融，审稿人很难相信不是「凑巧调参」。

4. **复现信息**  
   记录：模型名与版本、依赖版本、随机种子（若训练）、硬件概览、关键超参。本仓库脚本侧重推理与评测；若贡献在**训练**侧，需跟随所用框架（如官方 `transformers` 示例、社区微调仓库）单独整理一页 `REPRODUCE.md` 式说明——第十章「方向 2」里提到的 LoRA 等仍建议作为你自有仓库的延伸，而不是假定读者已在本教程里跑完全参训练。

5. **相关工作与伦理**  
   用 1～2 篇近作定位你的问题即可入门；数据若含人脸、证件、医疗等，需单独考虑合规与脱敏。教程不替代机构伦理审查要求。

**结论**：**有 Python 基础 + 有 idea 的读者，在补全上述「基线—对照—消融—复现」之后，完全可能把本教程里的脚手架升级成自己的工作并发文**；若只读完不做严格实验设计，则更可能停在「能做出 Demo、难写成论文」的阶段。若你以论文为目标，建议把第十章之前的产出（评测 JSONL、日志字段、Agent 工作流）当作**实验附录的雏形**，再专门投入 2～4 周做文献与对照实验，而不是期待单靠阅读正文自动达到顶会门槛。

更具体的**微调框架入口、JSONL 衔接、`eval_results.jsonl` 汇总脚本**见：[研究向补充：微调入口、数据衔接与评测汇总](../研究向补充-微调与实验记录.md)。
