# 第八章 构建一个图像问答 Demo

这一章把上一章的“命令行推理”升级成“可交互 Demo”。很多项目真正的起点都在这里：先把能力做成一个稳定、可演示、可反复验证的界面。

**衔接**：请求体字段尽量与 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 的 `eval_vlm_dataset.py` 习惯一致（同模型、同 `messages` 结构），这样 Demo 里复现的 bug 能直接回到评测集对照。复杂 OCR / 长图可在 Demo 稳定后翻 [扩展阅读](../扩展阅读.md)。

## 一、为什么建议先做 Demo 再谈复杂系统

一个简单 Demo 的价值往往被低估了。它至少能帮你同时验证四件事：

1. 模型接口是否稳定
2. 图像输入链路是否正确
3. 用户问题与输出格式是否合理
4. 这个能力是否真的值得继续做成产品

如果连一个小 Demo 都经常答错、答慢、答偏，那直接上复杂系统通常只会把问题放大。

## 二、本章代码说明

配套代码位于：

- `docs/chapter8/code/app.py`
- `docs/chapter8/code/requirements.txt`
- `docs/chapter8/code/.env.example`

这个 Demo 的思路是：

- 前端用 `Gradio`
- 后端通过 OpenAI 兼容接口请求多模态模型
- 用户上传图片并输入问题
- 返回模型答案
- 支持切换多种基础任务模式

## 三、安装依赖

```bash
pip install -r docs/chapter8/code/requirements.txt
```

## 四、配置环境变量

配置可以参考 `.env.example`：

```bash
OPENAI_BASE_URL=http://127.0.0.1:8000/v1
OPENAI_API_KEY=EMPTY
MODEL_ID=your-vlm
GRADIO_SERVER_NAME=127.0.0.1
GRADIO_SERVER_PORT=7860
DEFAULT_MODE=自由问答
```

## 五、运行 Demo

```bash
python docs/chapter8/code/app.py
```

运行后会启动本地页面：**页眉会显示当前 `OPENAI_BASE_URL` 与 `MODEL_ID`**，终端里也会打印同一行，改 `.env` 后记得重启进程。上传图片并输入问题即可，例如：

- 请描述这张图的主要内容
- 这张截图里报了什么错误
- 请提取图片中的关键信息
- 请用三点总结这张图的核心内容

这一版 Demo 额外支持四种基础模式：

- 自由问答
- 截图报错分析
- 关键信息提取
- 商品图卖点总结

页面里还会显示一个简单的运行状态，便于你区分：

- 未上传图片
- 正常完成
- 接口调用失败

## 六、这个 Demo 结构不复杂，但很典型

### 1. 上传图片

前端把图片路径传给后端。

### 2. 图片转 data URL

后端把图片编码为 base64，这样能通过标准 OpenAI 兼容消息格式发送。

### 3. 发送图文请求

消息结构通常是：

- 一段文本问题
- 一张图片

### 4. 返回结果

界面展示模型生成的答案。

这已经覆盖了大量多模态应用的最小主链路。

## 七、怎么把 Demo 往“可用产品原型”推进

当你确认这个最小版本能跑通后，可以继续加下面这些能力：

### 1. 输出模式切换

支持：

- 自由回答
- JSON 输出
- 关键信息抽取
- 摘要模式

### 2. 多图输入

允许用户上传多张图片，让模型做比较、归纳、前后对照。

### 3. 历史记录

保留上下文，做多轮视觉对话。

### 4. 任务模板

预设“截图报错分析”“商品图卖点提取”“文档字段抽取”等场景模板。

## 八、Demo 迭代时最值得加的不是特效，而是可观测性

优先记录这几项：

- 用户原始问题
- 图片文件名或摘要
- 模型输出
- 请求耗时
- 失败样例

这样你才能知道后续应该优化：

- prompt
- 模型
- 接口
- 还是图像预处理

如果你继续扩展，建议至少保留：

- 用户可见的状态提示
- 基础异常信息
- 默认任务模式
- 输出格式约束

## 九、一个实用的演示策略

如果你打算把这个仓库继续做成教程类项目，建议每个 Demo 都满足：

1. 一条命令能启动
2. 有清楚的环境变量说明
3. 能用默认问题直接演示
4. 支持继续接别的模型

这会让项目更像一个真正可复用的开源仓库，而不是一次性实验代码。

## 十、进阶实战：把 Demo 扩成场景应用

这一版 Demo 已经适合继续往“小应用”推进。最推荐的三个扩展方向是：

### 1. 截图报错分析

给问题模板加上：

- 错误定位
- 原因解释
- 下一步操作建议

### 2. 文档字段抽取

输出从自由文本改为 JSON，例如：

- 标题
- 日期
- 金额
- 关键结论

### 3. 商品图卖点生成

让模型先识别商品元素，再按电商文案格式生成卖点。

如果你不确定做哪个方向，建议先选“截图报错分析”，因为它最容易做出即时可感知的效果。

## 十一、章末练习

**动机**：Demo 把能力变成可演示、可反复验证的界面；练完你能让请求习惯与第五章评测一致，避免「脚本里对、界面里错」。

### 必做（约 20 分钟）

1. 为 Demo 新增一种任务模式，并设计对应提示词模板。

### 进阶（约 40 分钟）

1. 为 Demo 增加一个简单的耗时显示或错误提示。
2. 让 Demo 支持结构化输出，例如 JSON 或条目列表。

### 挑战（1～2 小时）

1. 选择一个场景，把 Demo 改成更像产品的原型（含最少一条用户可见的「失败时该怎么做」提示）。

## 十二、可选配图

Demo 章节建议配一张真实界面截图，再补一张任务模式说明图（自由问答、截图报错分析、关键信息提取、商品图卖点总结）。

## 十三、这一章想让你带走什么

多模态项目最适合的起步方式，往往不是先做训练，而是先做一个可交互 Demo。只要这一步做好了，你后面接数据、接 Agent、接服务编排都会顺很多。

**第九章**在单次问答之上加感知摘要、工具分支和简单规划；下面把 Demo 里已验证的接口，升级成可拆步骤、可降级的工作流。
