# 第五章 评测体系与工程选型

## 一、为什么排行榜不等于选型答案

很多学习者刚接触多模态时，最容易陷入的误区就是“找总分最高的模型”。但在真实场景里，模型选择通常取决于：

- 任务类型
- 响应速度
- 显存预算
- 中文表现
- OCR 能力
- 是否支持多图、多轮、长上下文
- 推理接口和部署难度

所以你真正需要的是一套评测框架，而不是一个单一榜单。

## 二、评测多模态模型时，至少要分五个维度

### 1. 感知能力

模型能不能看懂基本场景，识别关键对象和关系。

### 2. OCR / 文档理解能力

能不能读取截图、发票、PPT、网页、图表、公式、表格等复杂内容。

### 3. 推理能力

面对比较、排序、数数、图表归纳、多步问题时是否稳定。

### 4. 幻觉控制

当图中不存在某个元素时，模型会不会一本正经地编造。

### 5. 工程成本

包括：

- 显存和吞吐
- 首 token 延迟
- 服务稳定性
- 接口易用性

## 三、你可以自己搭一套最小评测集

对个人项目来说，不一定非要一开始就跑大型 benchmark。更实用的方式是先做一个场景内评测集。

例如，如果你要做“截图助手”，可以准备：

- 20 张网页截图
- 20 张 IDE 报错截图
- 20 张后台系统表单截图
- 20 张中文界面弹窗

然后设计一组固定问题：

- 这张图在说什么？
- 是否出现错误信息？
- 关键按钮是什么？
- 需要用户下一步做什么？

这样更贴近真实价值。

## 四、典型 benchmark 应该怎么看

公开 benchmark 的价值在于帮助你建立“能力分类”意识，而不是只看分数。

你可以把常见多模态 benchmark 粗分成：

- 综合理解类
- OCR / 文档类
- 数学与图表推理类
- 多学科推理类
- 幻觉与事实一致性类

评测时更值得关注的是：

1. 这个 benchmark 的输入分布是否接近你的场景？
2. 它更偏感知、OCR，还是偏推理？
3. 评测方式是生成式打分，还是固定答案匹配？

## 五、模型选型的一个实用原则：先看失败模式

与其只看“最好答案”，不如优先看模型怎么失败。因为很多业务问题都是在失败模式里暴露的。

建议重点观察：

- 遇到小字时是否直接忽略
- 遇到多对象时是否混淆
- 遇到复杂表格时是否漏列
- 遇到不存在的信息时是否编造
- 遇到中文 UI 时是否术语漂移

如果一个模型在你的关键场景里失败方式不可控，它就不适合作为主力方案。

## 七、面向场景的选型建议

### 1. 截图助手 / 界面理解

优先看：

- OCR 稳定性
- 中文界面理解
- 小字和图标识别
- 长图处理方式

### 2. 文档抽取

优先看：

- 版面理解
- 表格抽取
- 字段一致性
- JSON 输出稳定性

### 3. 电商内容生成

优先看：

- 商品图感知
- 生成文案的可控性
- 风格一致性
- 幻觉控制

### 4. 多模态 Agent

优先看：

- 多轮稳定性
- 对工具结果的融合能力
- 是否支持多图输入
- 延迟是否可接受

## 八、工程侧不要忽视的指标

很多教程只讲“准确率”，但工程里还有几项很关键：

- 首 token 延迟
- 平均响应时长
- 峰值显存
- 并发能力
- 图像上传和编码耗时
- 服务异常时的重试策略

这些指标往往直接决定系统能不能上线。

## 九、从学习者视角，如何做第一次模型比较

如果你现在就要开始实践，可以这样做：

1. 选 2 到 3 个你能实际调用的模型。
2. 自己做 30 到 50 条小型测试集。
3. 固定 prompt 和问题模板。
4. 比较输出质量、稳定性和速度。
5. 记录典型错误样例。

这套流程虽然朴素，但足够支撑个人项目的第一轮选型。

## 十、实战：跑一个最小评测脚本

为了让“评测意识”落地，仓库新增了这组文件：

- `docs/chapter5/code/eval_vlm_dataset.py`
- `docs/chapter5/code/sample_eval_dataset.jsonl`
- `docs/chapter5/code/requirements.txt`

它们的目标不是做大型 benchmark，而是帮你跑通“场景内小评测”。

### 你可以这样使用

1. 把 `sample_eval_dataset.jsonl` 改成你自己的图片和问题。
2. 配置 OpenAI 兼容接口地址与模型名称。
3. 运行 `eval_vlm_dataset.py` 批量请求模型。
4. 观察输出结果，并人工记录：
   - 是否答对
   - 是否答非所问
   - 是否出现幻觉
   - 哪类图片最容易失败

```bash
pip install -r docs/chapter5/code/requirements.txt
set OPENAI_BASE_URL=http://127.0.0.1:8000/v1
set OPENAI_API_KEY=EMPTY
set MODEL_ID=your-vlm
set DATASET_PATH=docs/chapter5/code/sample_eval_dataset.jsonl
set OUTPUT_PATH=docs/chapter5/code/eval_results.jsonl
python docs/chapter5/code/eval_vlm_dataset.py
```

### 为什么这一步重要

很多项目一开始就盯着大榜单，但忽略了自己真正会遇到的样本分布。这个小脚本能帮助你快速建立“先测自己场景”的习惯。

## 十三、本章小结

- 选多模态模型不能只看总榜分数，必须按任务拆能力。
- 先构建小而准的场景评测集，再决定模型和部署方案。
- 失败模式比单次高分更有参考价值。
